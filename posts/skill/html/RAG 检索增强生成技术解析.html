RAG 检索增强生成技术解析

发布日期：2026-04-16

RAG 检索增强生成技术解析

一、 背景与目标

大语言模型存在知识固化与幻觉问题，其局限性主要体现在：

  • 知识更新滞后（通常滞后数月）。
  • 不包含特定专业领域或企业私有数据。
RAG 正是为了解决这些问题而提出的技术架构，其核心目标是为大模型外接一个实时、可靠的知识库，从而实现“开卷考试”，增强回答的准确性与可靠性。

二、 核心原理

RAG 将传统 LLM 的“闭卷考试”转变为“开卷考试”，其基本流程为：

  1. 检索：根据用户问题，从外部知识库（如向量数据库）中检索最相关的文档片段。
  2. 增强：将检索到的上下文与原始问题结合，构建出信息更丰富的提示词。
  3. 生成：大模型基于包含参考资料的提示词，生成最终答案。

三、 标准实现流程

1. 离线阶段：构建知识库

目标：将原始文档转换为可被语义搜索的结构化知识库。

  • 步骤：
    1. 文档加载与解析：读取各种格式的原始文件。
    2. 文本分割 (Chunking)：将长文档切分为大小适中（如 512-1024 tokens）的文本块，块之间可设置重叠区域以保持语义连贯。
    3. 向量化 (Embedding)：使用 Embedding 模型（如 text-embedding-3-small）将文本块转换为语义向量。
  • 核心概念：向量在这里指的是一串能够表示文本语义的数字（例如 [0.1, -0.2, 0.8, ...]）。通俗地讲，它是机器为一段文本生成的、富含语义的“身份证号码”，语义相似的文本，其向量在数字空间中的位置也接近。简单来说，向量就是给内容编了一个“能反映含义的坐标”。
  • 核心概念：向量模型（即Embedding模型）是实现上述转换的工具。它的作用就像一个“翻译官”，将人类可读的文本“翻译”成机器可理解的向量（即“身份证号码”）。同一个模型能确保含义相近的句子（如“今天天气好”和“It's a sunny day”）被转换成相似的向量。简单来说，向量模型是“万物转坐标”的专用翻译官。
    1. 存入向量数据库：将向量与对应文本块存入数据库（如 Chroma, Milvus,Neo4j）。
  • 核心概念：向量数据库是一种专门为存储和检索向量而设计的数据库。与传统数据库按关键词查找不同，它擅长基于向量的相似性进行搜索。你可以把它想象成一个拥有“人脸识别”能力的超级图书管理员，能根据你提供的一段话的“意思”（向量），快速从海量资料中找到意思最接近的文档。简单来说，向量数据库是“按意思找东西”的超级索引柜。

2. 在线阶段：检索与生成

目标：响应用户查询，并返回基于上下文的答案。

  • 步骤：
  • 1. Query 向量化：将用户问题用相同的 Embedding 模型向量化。
  • 2. 相似度检索：在向量库中进行近似最近邻搜索，召回与问题最相关的 Top-K 个文本块。
  • 核心概念：相似度检索依赖于计算向量间的“距离”。常用方法有欧式距离和余弦相似度。
  • 欧式距离：衡量两个向量在多维空间中的“直线距离”。距离越近，数值越小，表示越相似。这好比比较两栋房子的实际空间距离。
  • 余弦相似度：衡量两个向量在方向上的差异，而忽略其长度。它关注的是“语义方向”是否一致。值越接近1，表示方向越一致，语义越相似。这好比比较两个人前进的方向是否相同，而不关心他们各自走了多远。在文本处理中，余弦相似度更为常用，因为它能更好地捕捉语义上的相似性。
  • 3. 构建 Prompt：将检索到的上下文填入预设模板，形成增强后的提示词。示例如下：
  •      请根据以下上下文回答用户的问题。如果上下文不包含相关信息，请如实告知。
  • 上下文：

    {context}

  • 用户问题：{question}
  • 回答：
  • 4. LLM 推理：将构造好的 Prompt 提交给大模型，生成最终答案。

四、 高级优化与工程要点

基础的 RAG 系统在生产环境中常面临“检索不准”或“上下文丢失”问题，以下为关键优化方向：

1. 提升检索质量

  • HyDE (假设性文档嵌入)：让大模型先根据抽象问题生成一个假设答案，再用此答案去检索，效果更佳。
  • 多路召回：结合向量检索与关键词检索，兼顾语义理解与精确匹配。
  • 重排序 (Reranker)：在召回结果后，使用专用模型（如 bge-reranker）对结果进行精细化重排，提升输入模型上下文的质量。

2. 优化上下文处理

  • 父文档检索：文档按粗细粒度分别切割。检索时找到最相关的细粒度片段，但将其所属的粗粒度文档整体作为上下文返回，以保持信息完整。
  • Agentic RAG：对于复杂多步查询，引入智能体，让模型自主决策检索、总结、二次检索等步骤。

3. 工程架构

  • 流式响应：支持流式输出答案，优化用户体验。
  • 引用来源：在答案中标注引用来源（如文件名、页码），增强可信度。

五、 核心价值总结

RAG 的本质是 “大模型外挂知识库”，它通过引入外部知识，有效缓解了大模型的知识滞后、幻觉及缺乏私有数据认知等核心痛点，是企业将大模型落地到具体业务场景（如知识库问答、智能客服）的主流方案。
