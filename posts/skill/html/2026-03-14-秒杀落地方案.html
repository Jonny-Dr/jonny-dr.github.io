高并发秒杀系统完整落地方案

一、核心设计思想与整体请求链路

核心设计原则

在原生产级方案基础上做定向升级：移除本地缓存以简化架构，强化前端防重、网关限流、前置过滤能力，遵循「前端拦无效、网关控流量、前置滤重复、原子扣库存、异步落数据」的分层削峰思路，兼顾高性能、防超卖与可维护性。

完整请求链路（流量走向）

用户点击秒杀 → 前端按钮置灰防重 → CDN静态加速 → Nginx负载均衡 → 网关层（Sentinel限流+非法脚本拦截） → 业务层布隆过滤器预过滤 → Redis乐观锁防重复提交 → Redis Lua脚本原子预扣库存 → MQ异步生成订单 → 数据库最终落库

二、各层详细实现方案

1. 前端层：强化交互防重+基础流量拦截

作为流量第一道拦截，低成本过滤90%以上无效请求：

  • 按钮强制置灰：用户点击秒杀按钮后立即置为不可点击状态，直至收到后端响应（成功/失败/超时）后恢复，从源头拦截用户频繁点击、重复提交产生的无效请求。
  • 页面静态化：秒杀活动页面全量静态化，部署至CDN节点，避免秒杀瞬间页面请求打向应用服务器。
  • 本地时间校验：前端本地校验秒杀活动时间，未开始、已结束的请求直接拦截，不调用后端接口。
  • 设备指纹采集：基于UA、设备标识生成唯一指纹，配合后端风控拦截批量脚本刷单。

2. 网关层：Sentinel 统一流量管控

替换原自研令牌桶限流，接入Sentinel组件实现生产级全维度流量管控：

  • 接口级限流：针对秒杀核心接口配置QPS阈值，支持令牌桶/漏桶算法，超出阈值直接快速失败，保护后端业务服务。
  • 多维度风控限流：支持按用户ID、IP地址、设备指纹三个维度设置请求频次阈值，拦截高频恶意刷单请求，对恶意来源直接封禁。
  • 熔断降级：监控后端服务响应耗时、异常比例，达到阈值自动熔断，返回友好降级提示，避免服务级联雪崩。
  • 系统自适应保护：基于服务器CPU使用率、系统负载动态调整限流阈值，保障服务整体稳定运行。

3. 业务前置过滤层：双层拦截过滤无效请求

请求到达业务服务后，先经过两层前置过滤，大幅减少无效请求对核心库存逻辑的冲击，降低Redis压力。

（1）布隆过滤器预过滤

  • 存储内容：两类数据——本次活动合法的秒杀商品ID集合、已抢购成功的「用户ID+商品ID」集合。
  • 校验逻辑：请求进入业务层后，先通过布隆过滤器校验：商品ID不存在、用户已抢购成功（判定不存在）则直接返回失败，不进入后续逻辑。
  • 定位：内存占用极低、查询性能O(1)，用于拦截非法商品请求、大部分重复抢购请求，属于粗粒度前置过滤。
  • 边界说明：最终一人一单校验以Lua脚本为准，规避布隆过滤器误判带来的体验问题。

（2）Redis乐观防重锁

  • 锁Key设计：seckill:lock:{goodsId}:{userId}，以商品ID+用户ID作为唯一锁标识。
  • 实现逻辑：通过Redis的SETNX命令尝试加锁，设置3~5秒短过期时间；加锁失败说明用户正在重复提交（网络重试、频繁点击），直接返回「请勿重复操作」。
  • 定位：拦截短时间内的并发重复请求，属于请求级防重；与Lua脚本的抢购结果级防重形成分层防护，减少无效请求执行Lua脚本的开销。

4. 核心库存层：Lua脚本原子预扣库存（防超卖核心）

经过前置多层过滤后，到达该层的均为有效请求。依托Redis单线程执行模型，通过Lua脚本一次性原子完成库存校验、扣减、用户记录全流程，彻底解决并发超卖问题，同时规避分布式锁的性能损耗。

（1）数据预热

秒杀活动开启前1-2小时，通过定时任务将秒杀商品库存、活动状态、价格等核心数据全量预热至Redis，并设置合理过期时间，避免高峰期缓存未命中导致请求穿透数据库。

（3）Java后端调用核心代码

@Service

public class SeckillServiceImpl implements SeckillService {



    @Autowired


    private StringRedisTemplate stringRedisTemplate;


    @Autowired


    private RocketMQTemplate rocketMQTemplate;



    // 加载Lua脚本


    private static final DefaultRedisScript<Long> SECKILL_SCRIPT;



    static {


        SECKILL_SCRIPT = new DefaultRedisScript<>();


        SECKILL_SCRIPT.setScriptText(ResourceUtil.readUtf8Str("seckill.lua"));


        SECKILL_SCRIPT.setResultType(Long.class);


    }



    @Override


    public Result seckill(Long goodsId, Long userId) {


        // ========== 前置1：布隆过滤器校验 ==========


        String bloomKey = "seckill:user:record:" + goodsId;


        boolean mightExist = stringRedisTemplate.opsForBloom().isExists(bloomKey, userId.toString());


        // 布隆过滤器判定不存在则一定未抢购，判定存在则继续后续校验，容忍小概率误判


        // 商品ID合法性校验逻辑同理，此处省略



        // ========== 前置2：Redis乐观防重锁 ==========


        String lockKey = "seckill:lock:" + goodsId + ":" + userId;


        Boolean lockSuccess = stringRedisTemplate.opsForValue()


                .setIfAbsent(lockKey, "1", 3, TimeUnit.SECONDS);


        if (Boolean.FALSE.equals(lockSuccess)) {


            return Result.fail("请勿重复操作");


        }



        try {


            // ========== 核心：执行Lua原子扣减 ==========


            String stockKey = "seckill:stock:" + goodsId;


            String userRecordKey = "seckill:user:" + goodsId;



            Long result = stringRedisTemplate.execute(SECKILL_SCRIPT,


                    Arrays.asList(stockKey, userRecordKey),


                    userId.toString());



            // 处理返回结果


            if (-1 == result) {


                return Result.fail("商品已抢空");


            }


            if (0 == result) {


                return Result.fail("请勿重复抢购");


            }



            // 抢购成功，发送MQ异步创建订单、落库


            SeckillOrderDTO orderDTO = new SeckillOrderDTO(goodsId, userId);


            rocketMQTemplate.syncSend("seckill_order_topic", 


                    MessageBuilder.withPayload(orderDTO).build());



            return Result.success("抢购成功，订单生成中");


        } finally {


            // 释放防重锁，也可等待自动过期，可根据业务场景选择


            stringRedisTemplate.delete(lockKey);


        }


    }


}

5. 异步削峰层：MQ解耦订单全流程

遵循「高峰期不操作数据库」的原则，所有耗时操作全部异步化。

  • 核心流程：Redis预扣库存成功后，立即向用户返回结果，同时发送订单消息至RocketMQ/Kafka；消费者异步消费消息，完成数据库订单创建、真实库存扣减、订单状态维护等操作。
  • 可靠性保障
  • 消息幂等性：基于订单唯一ID做幂等校验，避免重复消费创建重复订单。
  • 失败重试：消费失败开启自动重试，多次失败转入死信队列，人工兜底处理。
  • 超时关单：通过延迟消息实现15分钟未支付订单自动关闭，同步回补Redis与数据库库存。

6. 数据最终一致性保障

移除本地缓存后，所有热点数据统一存储于Redis，通过三层机制保证Redis缓存与数据库的最终一致性：

  1. 定时校验补偿：定时任务周期比对Redis缓存库存与数据库库存差异，自动校准修复数据偏差。
  2. 订单状态联动：未支付订单超时关闭后，同步回补Redis与数据库库存。
  3. 每日对账机制：每日凌晨执行秒杀全量数据对账，排查异常订单、库存差异，完成兜底修复。

三、方案核心特点总结

  1. 架构更简洁：移除本地缓存，消除多级缓存数据不一致问题，降低多节点运维复杂度，核心逻辑全部基于Redis+MQ实现，轻量化易落地。
  2. 拦截层级更丰富：形成「前端置灰→Sentinel限流→布隆过滤器→乐观锁→Lua脚本」五层拦截体系，无效请求逐层过滤，核心库存逻辑压力大幅降低，系统并发承载能力更强。
  3. 风控能力更成熟：基于Sentinel实现多维度限流、熔断降级，生产可用性高于自研限流，可灵活应对恶意刷单、流量突增等场景。
  4. 数据一致性有保障：Lua原子操作杜绝超卖，异步+定时补偿+对账三层兜底保证最终数据一致，无资损风险。
