# 有限资源并发抢占：生产级设计范式

> 本文档整理自一套成熟的**电商库存扣减与资源抢占**设计思想，旨在提供高并发下的稳定性与可靠性方案。

------

## 一、这套设计的核心思想本质

这不是一套单纯的「电商库存扣减代码」，而是一套**面向「有限资源并发抢占」场景的通用生产级设计范式**。

> 用「细粒度分布式锁」解决并发冲突，用「分层幂等+流水台账」保证操作可追溯、可回滚、防重复，用「缓存扛量+数据库兜底」平衡高并发性能与数据可靠性，最终实现**「高吞吐下不超发、不重复、可溯源、故障不雪崩」**。

**底层逻辑**：不相信任何单一组件100%可靠，通过层层设防、分层降级，在性能、一致性、可维护性三者之间找到生产环境最优解。

------

## 二、为什么要这么设计？（每个设计点的底层考量）

| 设计点                        | 核心痛点                                                     | 解决方案逻辑                                                 |
| ----------------------------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| **Redis + 分布式锁**          | 纯数据库扛不住高并发，易引发性能雪崩（行锁竞争、连接池打满）。 | **用缓存换性能**。Redis 单线程内存操作扛流量，数据库仅负责最终落账，压力降两个数量级。 |
| **双层幂等校验**              | 兼顾性能与并发正确性，避免“为了绝对正确牺牲所有性能”。       | **前置无锁查流水**（挡99%重复请求，无阻塞）+ **锁内二次幂等**（兜底极端并发）。单层设计要么性能浪费，要么并发失控。 |
| **流水表 (Ledger)**           | 无台账则无幂等、无回滚、无审计，系统变成黑盒，出问题必资损。 | **所有资源变更必须留痕**。流水表是幂等判断、回滚依据、对账基准的唯一真相源。 |
| **三分库存 (可用/锁定/已售)** | 直接减总数导致状态混乱，无法支持取消回滚，易引发账实不符。   | **状态隔离**。不同阶段的库存放在不同池子，通过“池间转移”实现清晰的生命周期管理（锁定→支付→释放）。 |
| **层层兜底机制**              | 分布式系统没有100%可靠的组件（Redis脑裂、DB切换、代码Bug）。 | **先考虑故障，再考虑正常**。四层防护（前置幂等→分布式锁→锁内幂等→DB唯一键）确保单点失效不影响全局。 |
| **SKU细粒度锁**               | 锁粒度越大，并发度越低（如订单锁导致全串行）。               | **最大化并发能力**。只有抢同一商品的请求互斥，不同商品互不影响；配合“按SKU排序”防止死锁。 |

------

## 三、这套设计思想的通用适用条件

只要业务满足以下 **3个特征**，均可套用：

1. **资源有限**：存在可量化的、不能超发的有限资源（商品、名额、额度、座位等）。
2. **并发抢占**：存在高并发下多个请求同时抢占同一份资源的场景。
3. **状态流转**：资源占用有完整的生命周期（锁定→确认→释放），需要支持取消、超时回滚，且操作需留痕可审计。

## 四、各行业的具体落地场景

### 1. 电商零售行业（最典型）

- **场景**：商品库存扣减、秒杀活动、优惠券发放、积分兑换。
- **映射**：SKU=有限资源，订单=占用凭证，库存流水=台账。

### 2. 本地生活与出行

- **酒店/民宿**：房间锁定 → 支付确认 → 入住核销。
- **影院/演出**：选座锁位 → 支付出票 → 超时释放。
- **打车/外卖**：发单 → 锁定骑手/司机 → 接单完成。

### 3. 金融与支付

- **额度管理**：贷款授信、支付额度、提现额度的锁定与扣减。
- **理财/基金**：募集期锁份额 → 成立后正式扣减。
- **抽奖活动**：奖品库存控制，防超发。

### 4. 票务与文旅

- **场景**：演唱会门票、高铁机票、景区门票。
- **特点**：峰值并发极高，依赖 Redis 扛压 + DB 兜底。

### 5. 企业服务与资源调度

- **云资源**：云服务器、带宽、存储的申请与锁定。
- **稀缺资源**：手机号、车牌、靓号的抢占。
- **预约挂号**：号源锁定与超时释放。

### 6. 营销与活动

- **场景**：活动名额报名、限量赠品、盲盒库存。
- **特点**：需支持独立库存池，与主库存隔离。

------

## 五、不适用的场景（边界说明）

| 场景类型                | 原因                                                         |
| ----------------------- | ------------------------------------------------------------ |
| **极限秒杀 (百万QPS+)** | SKU级锁仍有串行瓶颈，需用 Lua 脚本 + 库存分片 + 分桶扣减。   |
| **极低频后台操作**      | 无需引入分布式锁增加复杂度，直接操作数据库即可。             |
| **强一致金融交易**      | 如实时转账、证券交易，需数据库分布式事务，Redis 最终一致不满足。 |

------

## 六、总结

这套设计思想的核心价值，在于找到了**「高并发性能」**与**「数据可靠性」**之间的工业级平衡点：

- **稳健性**：非理论最强，但是生产中最稳妥、最好维护、性价比最高的方案。
- **普适性**：基于通用 Redis + MySQL，不依赖特殊中间件，适配绝大多数场景。
- **容错性**：任一组件故障不会导致资损或雪崩，适合双活、多机房部署。

这也是为什么几乎所有主流互联网公司的电商、出行、票务系统，核心资源抢占逻辑底层都是这套思路。
